Introduction

Welcome to the Nexus documentation. Here you'll find guides, API references, and best practices to help you get the most out of Nexus.

Nexus is a purpose-built issue tracker and sprint planner for modern engineering teams. It provides a lightning-fast interface, real-time synchronization, and deep integrations with your existing development workflow.

Just looking around? Check out our Quickstart Guide to get a project set up in under 5 minutes.

Quickstart

To get started with Nexus, you'll first need to create a workspace and invite your team. Once your workspace is set up, you can create your first project and start adding issues.

  1. Create a free workspace on app.nexus.io.
  2. Invite your team members via email or a shareable link.
  3. Connect your Git provider (GitHub, GitLab, or Bitbucket).
  4. Create your first project and define your issue types.

Installation (CLI)

For power users, we provide a CLI tool to interact with Nexus directly from your terminal. Install it via npm:

npm install -g @nexus/cli

Once installed, authenticate with your Nexus account:

nexus login
> Enter your API key: ********************************
> Successfully authenticated as jane@acme.com

Issues

Issues are the fundamental building blocks of Nexus. They represent tasks, bugs, features, or any other unit of work that needs to be tracked. Issues can be assigned to team members, labeled, prioritized, and linked to Git commits.

Issue Statuses

Nexus supports customizable workflows, but every project starts with a default set of statuses:

  • Backlog: Issues that are not yet ready to be worked on.
  • Todo: Issues that are ready to be picked up in the current sprint.
  • In Progress: Issues currently being worked on.
  • In Review: Issues that have an open Pull Request awaiting review.
  • Done: Completed issues.

Sprints

Sprints are time-boxed periods (usually 2 weeks) where a team commits to completing a set of issues. Nexus provides visual boards and burndown charts to help you track sprint progress and velocity.

Projects

Projects are used to group related issues and sprints. You can have multiple projects within a single workspace, and issues can be cross-linked between projects.

API Authentication

The Nexus API uses API keys for authentication. You can generate API keys in your workspace settings. All API requests must include your API key in the Authorization header.

curl https://api.nexus.io/v2/issues \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json"

Security Warning: Never expose your API key in client-side code or public repositories. Always use a backend proxy for API requests if building client applications.

Endpoints

The API is organized around REST. Our API has predictable resource-oriented URLs, accepts JSON-encoded request bodies, returns JSON-encoded responses, and uses standard HTTP response codes, authentication, and verbs.

  • GET /v2/issues - List all issues
  • POST /v2/issues - Create a new issue
  • GET /v2/issues/:id - Retrieve a specific issue
  • PATCH /v2/issues/:id - Update an issue

Webhooks

Webhooks allow you to receive real-time HTTP notifications when events happen in Nexus. You can configure webhooks to trigger on issue creation, status changes, sprint completions, and more.
